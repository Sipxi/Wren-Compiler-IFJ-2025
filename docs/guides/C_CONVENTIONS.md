# Стиль написания кода на C

Этот документ определяет стиль кодирования для нашего проекта. 
Соблюдение этих соглашений гарантирует, что наш код будет **читаемым, единообразным и поддерживаемым**.

---

## 🔤 Соглашения об именовании

* **Переменные**: `snake_case`

  ```c
  int buffer_size;
  char user_name[50];
  ```
* **Функции**: `snake_case`, форма глагол + существительное

  ```c
  void print_token();
  int parse_expression();
  ```
* **Константы и макросы**: `ALL_CAPS` с подчеркиванием

  ```c
  #define MAX_BUFFER 1024
  const int DEFAULT_PORT = 8080;
  ```
* **Типы (struct, enum, typedef)**: `PascalCase` или суффикс `_t`

  ```c
  typedef struct Token {
      int type;
      char *value;
  } Token;

  enum LogLevel { LOG_INFO, LOG_WARNING, LOG_ERROR };
  ```
* **Глобальные переменные**: префикс `g_`

  ```c
  int g_token_count;
  ```

---

## 📏 Форматирование

* **Отступы**: 4 пробела, без табов.
* **Длина строки**: максимум 100 символов.
* **Фигурные скобки**: стиль K\&R

  ```c
  if (x > 0) {
      do_something();
  } else {
      do_other();
  }
  ```
* **Пробелы**:

  * После ключевых слов: `if (x > 0)`
  * Вокруг операторов: `a + b`, не `a+b`
  * Звёздочка указателя относится к переменной, а не к типу

    ```c
    int *ptr;   // правильно
    int* ptr;   // избегать
    ```

---

## 🗂️ Файлы и модули

* Один `.c` + `.h` на модуль.
* Имена файлов: `snake_case.c`, `snake_case.h`
* **Защита заголовков** (или `#pragma once`):

  ```c
  #ifndef LEXER_H
  #define LEXER_H

  void tokenize(const char *source);

  #endif // LEXER_H
  ```
* **Порядок include**:

  1. Собственный заголовок
  2. Системные заголовки
  3. Сторонние заголовки

  ```c
  #include "lexer.h"
  #include <stdio.h>
  #include <stdlib.h>
  ```

---

## 📚 Функции и стиль кода

* **Одна задача на функцию** (желательно <50 строк).
* **Ранний возврат** вместо глубокой вложенности:

  ```c
  if (!user) return;
  ```
* Избегать **магических чисел** → использовать константы/макросы.
* **Обработка ошибок**:

  * Коды возврата (`0` = успех, `-1` = ошибка)
  * Использовать enum для детальных ошибок

---

## 📝 Комментарии и документация

* `//` для однострочных, `/* */` для блоков.
* Документировать каждую публичную функцию:

  ```c
  /**
   * Разбирает выражение из токенов.
   * @param tokens Список токенов.
   * @return Узел AST при успехе, NULL при ошибке.
   */
  ASTNode *parse_expression(TokenList *tokens);
  ```
* Писать комментарии о **почему**, а не очевидное **что**.


---

## 🛠️ Инструменты

* Использовать **clang-format** для единого стиля.
* Форматировать код перед пушем.
* Компиляторные предупреждения: собирать с `-Wall -Wextra -Werror`.
