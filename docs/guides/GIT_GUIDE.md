# 🐙 Git Гайд для нашего проекта

Привет, команда!  
Не переживайте, если вы никогда не работали с Git — тут всё описано максимально просто.  
Git — это система контроля версий. Она позволяет нам работать над одним проектом **вместе**, не мешая друг другу.

---

## 🚀 Самые важные команды (шпаргалка)

```bash
git clone <url>          # скопировать проект с GitHub
git checkout main         # переключиться на главную ветку
git pull                  # подтянуть последние изменения
git checkout -b имя/ветки # создать новую ветку и перейти в неё
git add .                 # добавить все изменения в коммит
git commit -m "msg"       # сохранить изменения (коммит)
git push origin ветка     # отправить ветку на GitHub
git status                # показать текущее состояние
git log                   # показать историю коммитов
````

---

## 🔑 Основные правила

1. У каждого будет своя **ветка по имени**:

   * `sipxi/feature-...`
   * `misha/feature-...`
   * `nika/feature-...`
   * `dima/feature-...`

   👉 Пример: если Sipxi делает лексер → `sipxi/lexer`.

2. В `main` **никто напрямую не пушит** 🚫.
   Только через Pull Request.

3. Коммиты должны быть маленькими и понятными:

   ```bash
   git commit -m "сделал разбор чисел в лексере"
   ```

---

## 📥 Первый раз (как начать работу с проектом)

1. Клонируем репозиторий:

   ```bash
   git clone <ссылка-на-репозиторий>
   cd <имя-папки>
   ```

2. Настраиваем имя и почту (один раз):

   ```bash
   git config --global user.name "ВашеИмя"
   git config --global user.email "почта@example.com"
   ```

## 🌱 Создание новой ветки для фичи

Каждый раз, когда начинаешь новую фичу, нужно создавать **свою отдельную ветку**, чтобы не мешать другим и main оставался чистым.

### 1. Переключаемся на `main` и подтягиваем обновления
```bash
git checkout main
git pull
````

### 2. Создаём новую ветку

```bash
git checkout -b <имя_ветки>
```

#### Примеры имён веток

* Sipxi делает лексер: `sipxi/lexer`
* Misha делает парсер: `misha/parser`
* Nika делает семантический анализ: `nika/semantic`
* Dima делает генератор кода: `dima/codegen`

> ⚡ Совет: ветку создаём **перед началом работы над фичей**, чтобы всё было организованно.

### 3. Проверяем, что мы на нужной ветке

```bash
git branch
```

* Текущая ветка будет отмечена звёздочкой `*`.

Теперь можно спокойно работать над фичей в своей ветке.

## 🏁 Каждый день перед работой

1. Переходим на `main`:

   ```bash
   git checkout main
   ```

2. Подтягиваем последние изменения:

   ```bash
   git pull
   ```

3. Переходим в свою ветку:

   ```bash
   git checkout sipxi/lexer
   ```

4. Обновляем ветку своей работой из `main` (если нужно):

   ```bash
   git merge main
   ```

---

## ✍️ Как работать в своей ветке

1. Делаем изменения в коде.
2. Добавляем изменённые файлы:

   ```bash
   git add .
   ```
3. Делаем коммит:

   ```bash
   git commit -m "описание что сделал"
   ```
4. Отправляем свою ветку на GitHub:

   ```bash
   git push origin sipxi/lexer
   ```

---

## ✅ Когда закончил работу на ветке

1. Пушим все изменения:

   ```bash
   git push
   ```

2. Заходим на GitHub → создаём **Pull Request (PR)** в `main`.

   * В названии указываем, что сделали.
   * Описываем изменения.

3. Команда смотрит PR, обсуждает и мержит в `main`. 🎉


## 🗑️ Удаление веток после завершения работы И МЕРДЖА В ГЛАВНУЮ

Когда фича закончена и Pull Request смержен в `main`, **ветку больше не нужно держать** — удаляем, чтобы не захламлять репозиторий.

### 1. Удаляем локальную ветку
```bash
git branch -d sipxi/lexer
````

* `-d` — удаляет ветку **только если все изменения влиты в main**
* Если вдруг ветка не влита, используйте осторожно `-D` (форсированное удаление)

### 2. Удаляем удалённую ветку на GitHub

```bash
git push origin --delete sipxi/lexer
```

* Это удалит ветку с GitHub, чтобы её там не оставалось

### ⚡ Полный порядок действий после завершения фичи

1. Убедиться, что PR смержен.
2. Удалить локальную ветку:

```bash
git branch -d <имя ветки>
```

3. Удалить удалённую ветку:

```bash
git push origin --delete <имя ветки>
```






## ❌ Чего НЕ делаем

* Не пушим в `main` напрямую.
* Не работаем все в одной ветке.
* Не коммитим гигантские изменения без описания.
